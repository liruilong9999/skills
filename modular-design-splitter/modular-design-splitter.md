# 模块化设计拆分 Skill（强制规则）

## Goal
基于输入的完整软件设计文档（SDD v1），将其安全、规范地拆分成多个体积可控的「模块详细设计文档」，便于不同AI或开发者并行开发。同时生成统一的拆分索引和开发规范，确保 traceability 和协作一致性。

---

## 1. 输入要求
必须输入一份完整的《软件设计文档_v1.md》（由 software-designer Skill 生成）。

---

## 2. 输出文件存放规范（强制）
所有输出文件必须按以下路径和命名规则存放：

- 拆分索引文件：`/docs/设计/模块/v1/模块拆分索引_v1.md`
- 各模块详细设计文件：`/docs/设计/模块/v1/01_模块名称_详细设计_v1.md`  
  （序号从 01 开始递增，模块名称使用中文简洁命名，例如 “01_设备接入模块_详细设计_v1.md”）

---

## 3. 输出内容结构（强制包含以下两部分）

### 第一部分：模块拆分索引（文件路径：/docs/设计/模块/v1/模块拆分索引_v1.md）

```markdown
# 模块拆分索引_v1

## 文档信息
| 项目 | 内容 |
|------|------|
| 文档名称 | 模块拆分索引_v1 |
| 拆分日期 | 当前日期 |
| 来源设计文档 | 《XXX软件设计文档_v1.md》 |
| 来源需求文档 | 《XXX需求文档_v1.md》 |
| 拆分模块总数 | X 个 |

## 拆分原则
1. 每个模块保持高内聚、低耦合，职责边界清晰。
2. 单个模块文档体积控制在合理范围，避免AI上下文超限。
3. 公共能力（安全、性能、部署等）只在索引中说明，各模块仅保留本模块相关内容并标注 traceability。
4. 模块编号采用 SD-xxx 格式，与原设计文档保持一致。

## 模块拆分总表
| 模块序号 | 模块编号 | 模块名称 | 对应原设计章节 | 主要职责 | 对应需求编号 | Git分支建议 | 文件路径 |
|----------|----------|----------|----------------|----------|--------------|-------------|----------|

## 公共规范引用
- 本项目统一使用《Git与代码开发规范_v1.md》
- 所有模块开发必须遵循该规范

## 后续开发建议
已将设计拆分为 X 个独立模块，可分别交给不同AI或开发者在各自Git分支上并行实现。
```

### 第二部分：每个模块的独立详细设计文档
每个模块生成独立文件，命名格式：/docs/设计/模块/v1/01_模块名称_详细设计_v1.md（序号递增）
每个模块详细设计文档强制模板：
```txt
# 【模块名称】详细设计_v1

## 文档信息
| 项目 | 内容 |
|------|------|
| 模块编号 | SD-xxx |
| 模块名称 | XXX模块 |
| 版本 | v1.0 |
| 拆分自 | 《XXX软件设计文档_v1.md》 |
| 对应需求文档 | 《XXX需求文档_v1.md》 |
| 文件路径 | /docs/设计/模块/v1/01_模块名称_详细设计_v1.md |

## 1. 模块概述
- 1.1 模块目标
- 1.2 职责边界（明确本模块做什么、不做什么）
- 1.3 与其他模块的依赖与接口关系

## 2. 功能与流程设计
- 核心处理流程（使用Mermaid绘制序列图或流程图）
- 子组件/子模块分解

## 3. 数据设计（仅本模块相关）
- 本模块涉及的数据实体、表结构、缓存设计
- 数据流转说明

## 4. 接口设计（仅本模块相关）
- 本模块对外提供的接口
- 本模块依赖的外部接口（注明调用方式）

## 5. 实现要点与技术方案
- 关键业务逻辑与算法说明
- 性能、异常、安全相关在本模块的具体实现路径
- 与原设计文档对应章节的traceability

## 6. 测试要点
- 单元测试覆盖范围
- 集成测试关注点

## 7. 验收标准
- 本模块完成后的可验证指标

## 8. 注意事项与风险
- 本模块开发过程中的重点风险及应对

## 附录
- 相关术语说明
```


## 4. 统一的 Git 与代码开发规范（必须生成）

文件路径建议：/docs/设计/模块/v1/Git与代码开发规范_v1.md
内容必须包含以下强制章节：

```txt
# Git与代码开发规范_v1

## 1. Git 分支规范（强制）
- 主分支：main
- 开发集成分支：develop
- 功能分支命名规则：feature/module-[模块编号]-[模块英文简写]
  示例：feature/module-SD-301-realtime-collector
- 每个模块必须使用独立分支开发，禁止在同一分支修改多个模块
- 合并流程：feature分支 → develop → main（需提交PR并经过Review）

## 2. 代码开发通用规范
- 严格遵循项目技术选型（语言、框架、版本）
- 目录结构按模块组织
- 命名规范：采用驼峰/下划线一致风格
- 注释要求：公共接口必须有清晰中文注释
- 日志规范：统一使用结构化日志
- 异常处理：必须捕获并记录，不允许吞异常
- 单元测试：核心逻辑覆盖率不低于70%
- Commit Message 格式：type(scope): subject （例如 feat(SD-301): 实现实时订阅引擎）

## 3. 多人并行开发规则
- 每个模块负责人仅修改自己的 feature 分支
- 公共基础组件的修改必须通过PR提交到 develop 分支
- 跨模块接口变更必须同步更新模块拆分索引文档
- 开发完成后在PR描述中注明已完成的验收项

## 4. 代码提交 checklist
- [ ] 符合分支命名规范
- [ ] 通过本地单元测试
- [ ] 更新了本模块详细设计文档中的实现说明
- [ ] 添加了必要的中文注释
- [ ] Commit Message 格式正确
```

## 5. Enforcement（强制执行规则）

- 拆分必须完整覆盖原设计文档所有功能模块，不得遗漏
- 每个模块文档必须保留与原设计文档和需求文档的 traceability
- 公共内容（整体安全、部署等）不得在各模块中重复描述，只在索引中引用
- 输出时必须明确说明文件存放路径
- 拆分后模块数量建议控制在 6~12 个之间
- 生成完成后，在模块拆分索引文档末尾添加提示：“本设计已拆分成 X 个独立模块，可分别输入给不同AI或开发者进行代码实现。”

## 使用说明

- 将完整的软件设计文档作为输入提供给本 Skill。
- 本 Skill 将自动按指定路径和命名规则输出：
- 模块拆分索引_v1.md_
- _多个 01_模块名称_详细设计_v1.md 文件
- Git与代码开发规范_v1.md
- 版本根据修改情况改为v2、v3等

后续可将每个模块的详细设计文档单独喂给不同AI，让其基于该文档进行代码开发。
