# 软件设计文档生成规则（强制）

## Goal
基于输入的结构化需求文档，生成一份专业、严谨、可直接指导开发、测试和运维的**软件设计文档（Software Design Document，SDD）**。

文档风格：工程化、结构化、表格化为主，辅以必要的架构图、流程图和序列图。

---

## 1. 语言与表达规则
- 全部使用中文描述（技术术语、协议、代码片段除外）
- 表达严谨、专业，避免口语化
- 重要内容必须使用表格呈现
- 每个设计元素需明确 traceability 到对应的需求编号（FR-、NFR- 等）

---

## 2. 设计方法（必须在文档中声明）
默认采用 **结构化设计 + 模块化分层架构** 方法：
- 自顶向下进行系统分解
- 明确定义分层架构
- 强调高内聚、低耦合、可扩展性、可维护性
- 在文档开头明确说明采用的设计方法及理由

---

## 3. 输出文档结构（强制严格遵循以下章节顺序）

### 文档信息（表格）
| 项目          | 内容 |
|---------------|------|
| 文档名称      | XXX系统软件设计文档 |
| 文档版本      | v1.0 |
| 设计日期      | YYYY-MM-DD |
| 输入来源      | 《XXX需求文档 vX.X》 |
| 设计方法      | 结构化设计 + 模块化分层架构 |
| 目标读者      | 架构师、开发人员、测试人员、运维人员 |

### 1. 设计概述
- 1.1 设计目标（简要对应需求单一目标）
- 1.2 设计约束与原则（对应主要非功能需求）
- 1.3 系统范围与边界
- 1.4 设计假设与依赖

### 2. 系统架构设计
- 2.1 整体架构图（使用 Mermaid 绘制）
- 2.2 分层架构说明（推荐至少包含：表现层、应用服务层、领域/业务服务层、数据接入/持久化层、基础设施层）
- 2.3 部署架构视图（支持单机、容器化、集群等常见方式）
- 2.4 技术选型说明（列出主要技术组件及选择理由）

### 3. 模块分解与详细设计
按需求中的主要功能模块进行分解，每个模块包含：
- 模块编号（SD-xxx）与名称
- 模块职责与目标
- 与需求的 traceability（对应 FR-/NFR-）
- 输入/输出
- 子模块/组件分解
- 关键流程（Mermaid 序列图或流程图）
- 模块间接口关系

### 4. 数据设计
- 4.1 核心数据模型（ER 图或类图，使用 Mermaid）
- 4.2 数据库设计（主要表结构、索引、分区/保留策略）
- 4.3 数据字典（关键实体说明）
- 4.4 数据流转与一致性设计

### 5. 接口设计
- 5.1 外部接口（南向/北向接口）
- 5.2 内部服务接口（API 定义或消息接口）
- 5.3 接口规范（请求响应格式、错误码、版本控制）
- 5.4 接口调用关系图（可选序列图）

### 6. 用户界面设计
- 6.1 整体 UI 风格与设计原则
- 6.2 主要页面布局与交互流程描述
- 6.3 关键用户操作流程（Mermaid 流程图）

### 7. 安全设计
- 7.1 认证与授权模型
- 7.2 数据传输与存储安全
- 7.3 权限控制粒度
- 7.4 安全审计与日志
- 7.5 安全风险防护措施

### 8. 性能与可靠性设计
- 8.1 性能指标分解与实现方案（对应需求中的量化指标）
- 8.2 并发、缓存、异步、削峰填谷等策略
- 8.3 高可用、容错、重试、降级机制
- 8.4 监控与告警设计

### 9. 部署与运维设计
- 9.1 部署模型（单机、容器、集群）
- 9.2 配置管理方式
- 9.3 日志、监控、健康检查机制
- 9.4 备份、恢复与升级策略

### 10. 扩展性设计
- 10.1 插件化/模块化扩展机制
- 10.2 未来功能预留接口与抽象层设计

### 11. 异常处理与容错设计
- 对应需求中异常与边界情况的处理策略

### 12. 主要设计决策与权衡
- 关键技术/架构决策说明
- 备选方案对比（如果适用）

### 13. 风险与未决项
- 技术风险清单及缓解措施
- 待确认的设计项

### 14. 附录
- 术语表
- 参考文档
- 版本修改历史

---

## 4. 输出质量要求（强制）
- 文档结构必须完整，不得缺失任何章节
- 至少包含 3 张 Mermaid 图（整体架构图、核心业务流程图、数据模型图）
- 每个模块和重要设计点必须标注与需求的对应关系（traceability）
- 性能指标必须有明确的实现路径，而非仅描述
- 输出文件建议命名为：`[系统名称]软件设计文档_v1.md`
- 输出文件放在`/docs/设计/v1(需求_v1).md`，必须对应需求版本
- 设计深度应达到可直接进入编码实现阶段（提供足够细节，但不写入具体代码实现）

---

## 5. Enforcement
- 输入必须为结构化需求文档
- 输出必须严格按上述章节顺序组织内容
- 如需求中存在模糊项或假设，设计文档中需明确说明“基于需求文档中 XX 假设进行设计”
- 生成完成后，可在文档末尾建议用户对关键设计决策进行确认，以便输出 v2 版本
