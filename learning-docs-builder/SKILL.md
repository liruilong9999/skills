# Learning Docs Builder

## 目标

在当前工作区的 `docs/` 目录下，为指定主题生成一套从入门到精通的中文学习资料，而不是只输出零散说明。

优先产出可持续学习的资料体系，包含学习顺序、核心概念、示例、练习、进阶专题、常见误区和精通路线。

## 输出位置

- 默认在当前项目下创建 `docs/学习/<主题>/`
- 如果用户明确要求直接写入 `docs/` 根目录，则遵循用户要求
- 如果项目已经存在同主题学习资料，则优先增量更新，不重复新建平行目录

## 何时直接触发

在以下请求中直接使用本 Skill：

- 用户要求“帮我学某个技术”
- 用户要求“生成从入门到精通的学习资料”
- 用户要求“整理知识体系”“做教程文档”“输出学习路线图”
- 用户要求“在当前项目 docs 下生成学习文档”
- 用户希望围绕某个框架、协议、语言、系统或代码库沉淀项目内学习资料

典型触发示例：

- “帮我在当前项目 docs 下生成一套 OPC UA 学习资料”
- “做一套 Vue 3 从入门到精通教程”
- “把这个项目涉及的 Kafka 知识整理成学习文档”

## 执行流程

### 1. 明确主题与目标读者

- 优先使用用户明确给出的主题
- 如果主题不完整，则结合当前工作区中的代码、文档、目录名和用户上下文进行推断
- 如果存在多个高风险候选主题，再提出一个简短澄清问题；否则直接做合理假设并在结果中说明
- 默认读者为“希望系统学习该主题的工程实践者”，默认目标为“从入门到可独立落地”

### 2. 建立最小上下文

在生成资料前，先快速检查当前项目中的以下信息：

- `docs/` 中是否已有相关文档
- `README`、需求文档、设计文档、示例代码中是否出现目标主题
- 当前项目是否更偏概念学习、工程实践、源码阅读或企业落地

只读取完成任务所需的最小上下文，避免无意义全仓遍历。

### 3. 设计学习资料结构

默认生成一套分阶段学习资料。除非用户明确要求精简版本，否则至少包含以下文件：

- `00-学习导航.md`
- `01-入门总览.md`
- `02-核心概念.md`
- `03-基础实践.md`
- `04-关键机制与常用模式.md`
- `05-进阶专题.md`
- `06-工程实践与案例.md`
- `07-常见问题与误区.md`
- `08-精通路线图.md`

必要时补充以下可选文件：

- `09-练习题与参考答案.md`
- `10-术语表.md`
- `11-源码阅读指南.md`
- `12-面试题与讨论题.md`

设计结构时，读取 [references/learning-structure.md](references/learning-structure.md)。

### 4. 生成文档内容

每篇文档都要围绕“可学习、可实践、可继续深入”来写，不要写成百科词条堆砌。

每篇文档默认包含以下模块，可按主题裁剪但不要缺少核心教学闭环：

- 本篇目标
- 先修知识
- 核心内容
- 示例或类比
- 最小实践
- 自测题或检查清单
- 下一步学习建议

### 5. 保持系列化与可导航

- 在 `00-学习导航.md` 中给出完整阅读顺序
- 标注每篇文档解决什么问题
- 标注建议阅读时长、适合读者和进入下一篇的条件
- 如果主题跨度大，拆成“基础线”“工程线”“深入线”三条路径

## 内容规则

### 语言规则

- 正文默认使用中文
- 如果首次出现专业术语，使用“英文(中文解释)”格式，例如：`message broker(消息代理)`、`throughput(吞吐量)`
- 图表、流程图、表格标题优先使用中文

### 教学规则

- 先讲全景，再讲细节
- 先给直觉，再给定义
- 先给最小可运行示例，再延伸到复杂场景
- 解释“为什么这样设计”，不要只罗列“是什么”
- 对容易混淆的概念主动做对比

### 工程规则

- 内容要尽量贴近当前项目语境
- 如果当前项目已经暴露出真实案例，优先把案例写进文档
- 如果缺乏项目上下文，则使用行业通用案例，并明确说明是假设示例
- 不要覆盖用户已有文档；若更新现有学习资料，先保留原有结构并增量补强

## 输出质量门槛

生成结果必须满足以下要求：

- 目录结构清晰，读者能按顺序学习
- 每篇文档有独立价值，同时与前后篇衔接
- 至少包含概念、实践、进阶三个层级
- 不只讲 API 或语法，还要包含适用场景、边界、常见坑和实践建议
- 不出现“后续可补充”“这里略”等未完成占位内容

## 结果汇报

完成后，向用户简要汇报：

- 创建或更新了哪些学习文档
- 学习资料覆盖了哪些阶段
- 如果做了主题或路径假设，明确说明假设内容
